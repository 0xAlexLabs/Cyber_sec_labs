# 📘 Hack The Box: Appointment

> 🔗 https://app.hackthebox.com/machines/Appointment

---

## 🎯 Цель

Получить доступ к веб-приложению и извлечь флаг, используя уязвимость в механизме аутентификации.

---

## 🧠 Ключевая идея

Эта лабораторная работа демонстрирует классическую веб-уязвимость — **SQL Injection**, которая позволяет обойти форму входа без знания корректного пароля.

Главная идея машины:

- на хосте открыт только веб-сервис;
- основной вектор атаки — форма логина;
- пользовательский ввод небезопасно обрабатывается на стороне сервера;
- инъекция изменяет логику SQL-запроса;
- это приводит к **authentication bypass**.

Иными словами, здесь не нужно подбирать пароль или искать сложный эксплойт — нужно правильно понять, **как приложение проверяет учётные данные**, и использовать это против него.

---

## 🔍 Ход решения

### Шаг 1 — Сканирование цели

```bash
nmap -sC -sV <TARGET_IP>
```

### Почему это работает

`nmap` позволяет определить:
- какие порты открыты;
- какие сервисы доступны;
- какие версии сервисов используются.

Флаг `-sC` запускает стандартные NSE-скрипты.  
Флаг `-sV` определяет версию сервиса.

### Что мы узнаём

Результат сканирования показывает:

- хост доступен;
- открыт только **80/tcp**;
- сервис: **Apache httpd 2.4.38 ((Debian))**;
- заголовок страницы: **Login**.

### Почему это важно

Это ключевой этап мышления пентестера:

- SSH нет;
- SMB нет;
- FTP нет;
- значит, **единственная поверхность атаки — веб-приложение**.

После такого `nmap` нет смысла распыляться на другие сервисы: всё внимание должно идти в HTTP.

---

### Шаг 2 — Первичный анализ веб-приложения

После открытия сайта в браузере становится видно простую форму входа.

На странице присутствуют:

- поле `username`;
- поле `password`;
- чекбокс `remember me`.

### Почему это важно

Форма аутентификации — это не просто интерфейс, а логическая точка проверки доступа.

Для атакующего это означает:

- приложение принимает ввод пользователя;
- сервер сравнивает этот ввод с данными в базе;
- если ввод вставляется в SQL-запрос небезопасно, можно изменить логику проверки.

На данном этапе правильная мысль не “брутить директории”, а задать себе вопрос:

> Как именно сервер проверяет логин и пароль?

---

### Шаг 3 — Формирование гипотезы об SQL Injection

Типичная небезопасная логика на сервере может выглядеть примерно так:

```sql
SELECT * FROM users
WHERE username = 'admin' AND password = '123456';
```

Если разработчик строит запрос через конкатенацию строк, пользовательский ввод становится частью SQL-кода.

### Почему это опасно

Если атакующий контролирует содержимое полей формы, он может:

- закрыть исходную строку кавычкой;
- добавить собственное логическое условие;
- закомментировать остаток запроса.

Именно это и лежит в основе **SQL Injection**.

### К какому классу относится уязвимость

С точки зрения OWASP Top 10 (2021), это:

> **A03:2021 – Injection**

---

### Шаг 4 — Эксплуатация уязвимости

В поле `username` был введён payload:

```sql
' OR 1=1 -- 
```

В поле `password` можно ввести любое значение.

### Почему это работает

Предположим, приложение строит запрос так:

```sql
SELECT * FROM users
WHERE username = '<username>' AND password = '<password>';
```

После подстановки инъекции запрос логически превращается в:

```sql
SELECT * FROM users
WHERE username = '' OR 1=1 -- ' AND password = 'anything';
```

Разберём по частям:

#### 1. `'`
Закрывает исходную строку в SQL-запросе.

#### 2. `OR 1=1`
Добавляет условие, которое **всегда истинно**.

#### 3. `--`
Превращает остаток строки в комментарий.

То есть фрагмент:

```sql
' AND password = 'anything'
```

уже не обрабатывается базой данных.

### Итог

Условие `WHERE` становится истинным независимо от пароля, и приложение считает проверку успешной.

Это и есть **authentication bypass** — обход аутентификации без знания валидных учётных данных.

---

### Шаг 5 — Альтернативный вариант через комментарий для admin

В одном из заданий также требуется войти как `admin`, используя комментарий.

Для этого можно использовать payload вида:

```sql
admin' -- 
```

### Почему это работает

Такой ввод приводит запрос к логике:

```sql
SELECT * FROM users
WHERE username = 'admin' -- ' AND password = '...';
```

Всё, что идёт после `--`, игнорируется.  
Проверка пароля выключается, и приложение проверяет только имя пользователя `admin`.

Основная проблема приложения заключается в том, что пользовательский ввод вставляется в SQL-запрос без безопасной обработки.

---

## 🛡️ Рекомендации

Чтобы защититься от подобных атак, необходимо:

- использовать **prepared statements / parameterized queries**;
- не строить SQL-запросы через конкатенацию строк;
- валидировать и фильтровать ввод;
- ограничить права аккаунта базы данных;
- не выводить подробные ошибки SQL пользователю;
- внедрять WAF и базовые механизмы защиты веб-приложений.

## 📚 Чему учит эта машина

Appointment — очень простая, но методологически важная машина.

Она учит:

- не игнорировать результаты `nmap`;
- связывать порт с типом сервиса и возможным вектором атаки;
- видеть в форме логина потенциальную поверхность атаки;
- понимать SQL Injection не как “магическую строку”, а как изменение логики SQL-запроса;
- документировать не только шаги, но и ход мысли.

Именно такие машины хорошо формируют фундамент для дальнейшего веб-пентеста.
