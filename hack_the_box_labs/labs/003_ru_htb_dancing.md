# 📘 Hack The Box: Dancing

> 🔗 https://app.hackthebox.com/machines/Dancing

---

## 🎯 Цель

Получить доступ к SMB-шаре и извлечь флаг.

---

## 🧠 Ключевая идея

Эта лабораторная работа показывает типичную ошибку конфигурации SMB:
- сервис SMB доступен по сети;
- одна из шар разрешает доступ без валидных учётных данных;
- из-за этого можно перечислить содержимое и скачать файлы.

Главная идея здесь не в эксплойте, а в **enumeration + abuse of misconfiguration**.

---

## 🔍 Ход решения

### Шаг 1 — Сканирование цели

```bash
nmap -sV <TARGET_IP>
```

### Почему это работает

`nmap` отправляет сетевые запросы к целевой машине и определяет:
- какие порты открыты;
- какие сервисы слушают эти порты;
- иногда — версию сервиса.

Флаг `-sV` включает определение версии сервиса. Это важно, потому что нам недостаточно знать, что порт просто открыт: нужно понять, **что именно работает на нём**.

### Что мы узнаём

Результат показывает:
- `445/tcp open`
- сервис `microsoft-ds`

### Почему это важно

Порт **445** — это стандартный TCP-порт для SMB.  
Если он открыт, это означает, что на машине доступен файловый сетевой сервис Windows/Samba, и его нужно перечислять.

---

### Шаг 2 — Enumeration SMB-шар

```bash
smbclient -L //<TARGET_IP> -N
```

### Почему это работает

`smbclient` — это клиент для взаимодействия с SMB-ресурсами.

Разбор команды:
- `-L` — запросить список доступных SMB-шар;
- `//<TARGET_IP>` — указать удалённый SMB-хост;
- `-N` — не запрашивать пароль, то есть попробовать подключение без аутентификации.

Когда сервер допускает guest/anonymous-доступ, он отвечает списком доступных шар даже без логина и пароля.

### Что мы получаем

Обычно видим такие шары:
- `ADMIN$`
- `C$`
- `IPC$`
- `WorkShares`

### Почему это важно

Первые три — стандартные системные SMB-ресурсы:
- `ADMIN$` — административная шара;
- `C$` — скрытый доступ к системному диску;
- `IPC$` — служебная IPC-шара.

Они обычно требуют привилегий.  
А вот **`WorkShares`** выглядит как пользовательская/кастомная шара, а именно такие шары чаще всего и бывают настроены неправильно.

---

### Шаг 3 — Подключение к доступной шаре

```bash
smbclient //<TARGET_IP>/WorkShares -N
```

### Почему это работает

Здесь мы уже не просто запрашиваем список шар, а подключаемся к конкретной шаре `WorkShares`.

Если на этой шаре разрешён анонимный доступ, сервер пускает нас внутрь и выдаёт интерактивную SMB-консоль:

```text
smb: \>
```

### Почему мы идём именно в WorkShares

Потому что:
- `ADMIN$` и `C$` почти всегда закрыты для обычных пользователей;
- `IPC$` не предназначена для обычного просмотра файлов;
- `WorkShares` — единственная логичная цель для поиска полезных данных.

---

### Шаг 4 — Просмотр содержимого

```bash
ls
```

### Почему это работает

После успешного подключения мы уже работаем **внутри удалённой SMB-шары**, почти как в файловой системе.

Команда `ls` показывает содержимое текущего каталога на удалённой машине.  
Обычно здесь находятся директории:

- `Amy.J`
- `James.P`

### Почему это важно

Это указывает, что шара используется как рабочее файловое хранилище для пользователей.  
Значит, внутри могут лежать:
- заметки;
- конфигурационные файлы;
- учётные данные;
- флаг.

Именно поэтому на этапе enumeration всегда нужно **не только найти шару, но и пройтись по её структуре**.

---

### Шаг 5 — Переход по каталогам и извлечение файлов

```bash
cd Amy.J
get worknotes.txt
cd ..
cd James.P
get flag.txt
```

### Почему это работает

Команды внутри SMB-консоли очень похожи на обычные shell-команды:

- `cd` — перейти в каталог;
- `get` — скачать файл с удалённой SMB-шары на локальную машину;
- `cd ..` — вернуться на уровень выше.

### Почему сначала смотрим `Amy.J`

Потому что в пентесте полезно собирать не только конечный артефакт, но и **контекст**.  
Файл `worknotes.txt` может содержать:
- подсказки по следующим сервисам;
- имена пользователей;
- внутренние заметки;
- следы плохой operational security.

Даже если он не нужен для текущего флага, это правильная методика: **собирать интересные артефакты, а не прыгать сразу к цели**.

### Почему затем идём в `James.P`

Потому что после проверки одной пользовательской директории логично проверить и вторую.  
Внутри находится `flag.txt`, а это и есть целевой артефакт лабораторной работы.

---

### Шаг 6 — Чтение флага локально

```bash
cat flag.txt
```

### Почему это работает

Команда `get flag.txt` уже скачала файл на твою локальную машину.  
После этого `cat` просто выводит содержимое локального файла в терминал.

## 💥 В чём уязвимость

Здесь используется не memory corruption и не RCE, а **ошибка конфигурации**:

- SMB-сервис доступен по сети;
- на кастомной шаре разрешён доступ без нормальной аутентификации;
- пользователи хранят чувствительные файлы в доступной шаре.

Это очень реалистичный сценарий для внутреннего пентеста: неэксплуатируемая на первый взгляд служба оказывается источником утечки данных из-за слабых настроек доступа.

---

## 🛡️ Рекомендации

- отключить anonymous/guest access на SMB-шарах;
- ограничить доступ только авторизованным пользователям и группам;
- не хранить чувствительные файлы в общих шарах без ACL;
- регулярно проводить аудит доступных SMB-ресурсов;
- логировать доступ к сетевым файлам и отслеживать аномалии.
